Time tide is a new programming language

  1. It allows building finite state machines.
  2. It allows developing synchronous concurrent programs.
  3. It allows for shared memory or distributed memory concurrency.
  4. The compiler produces efficient C++ code, which allows for easy interaction with native code and deployment on embedded platforms.

Requirements when reading this document

  1. The reader is expected to have knowledge of:
    • Synchronous programming principles
    • Good knowledge of C/C++ including template meta-programming
  2. Please read the comments in the code snippet along with the text in this document.
  3. Time tide language and tool suite is a work in progress – expect bugs and things to change daily
  4. Please file bug reports on github repository with code snippets when things do not work.

Installing the Time tide compiler(s)

  1. Time tide uses multiple compiler tool chains.
  2. You are expected to have installed C++ compilers with support for at least C++-20 standard.
  3. Pull the Time tide compiler from: https://github.com/PRETgroup/timetide
  4. Install the Ocaml programming language compiler from here: https://ocaml.org/install#linux_mac_bsd
  5. Install dune with command opam install dune
  6. Now go to the timetide directory and do command dune build
  7. It will not compile and list the missing libraries.
  8. You will need to manually install the libraries like so: opam install <lib-name>
  9. You will need to install the following libraries: pretty_expressive ppx_deriving ppx_sexp_conv sexplib base
  10. Once all the libraries are installed you should be able to build the compiler using dune build in the timetide directory.
  11. Download and install the Rust programming language compiler from: Rust
  12. Pull the Sysrust intermediate language compiler from: https://github.com/amal029/sysrust
  13. Go inside the Sysrust compiler directory and run: make br
  14. Download and pull the Time tide example directory from: https://github.com/PRETgroup/Timetide_tests

  15. Go inside the Timetide_tests directory and run the following command: 

    COMPILE_SWITCH=sysrs FNAME="abro" SYSRUST_DIR=<path-of-sysrust-compiler-dir> \
                     TIMETIDE_DIR=<path-of-timetide-compiler-dir> make cleanall
  16. In the above command replace the <path-of-sysrust-compiler-dir> with where you have downloaded the individual timetide and sysrust compilers.

  17. Now run the command: 

    COMPILE_SWITCH=sysrs FNAME="abro" \
                     SYSRUST_DIR=<path-of-sysrust-compiler-dir> \
                     TIMETIDE_DIR=<path-of-timetide-compiler-dir> compilers
    • This will build all the compilers.

Compiling first example program

We will use the abro.tt program in Timetide_tests directory as our first program. 

 1: main() {
 2:   int channel A delay 1;
 3:   int channel B delay 1;
 4:   int channel R delay 2;
 5:   int channel O delay 1;
 6:   {
 7:     for (int i:1:99) {
 8:       abort(R) {                /* preemption when channel has data */
 9:         /* Wait for A and B in parallel */
10:         {run waitA(A);}<>{run waitB(B);};
11:         send O(100);            /* send O once both are received */
12:         /* abort when R is present and restart */
13:         task(period=1,duration=0,offset=0){;} /* halt */
14:       }
15:     } /* loop back */
16:   }<> /* synchronous parallel concurrency */
17:     {/*The environment*/
18:       {send A(1);}<>{send B(1);};
19:     }
20: };
21: /* Other functions/reactions being used inside main */
22: waitA(input int A) {abort(A) {task(period=1,duration=0,offset=0){;}}};
23: waitB(input int B) {abort(B) {task(period=1,duration=0,offset=0){;}}}
  1. The code above is a shared memory synchronous program.
  2. The program' entry point is main.
  3. The program has three modules/functions/reactions: waitA, waitB, and main.
  4. The main function waits for some data to be available on channels A and B in synchronous parallel.
  5. Once the channel data is received; main sends data on channel O.
  6. If during the process of waiting if any data is available on channel R, main pre-empts waiting for A and B and the program is reset.
  7. Channels are typed and have an associated delay.
  8. In the abro.tt program sending data in channel A or B makes it available after a single so called tick.
  9. Sending data in channel R makes it available after two ticks.
  10. The concept of a tick is central to programming in Time tide.
  11. Tick is the synchronous (logical) clock running the whole program.

Compiling the abro (shared memory) program

  1. Run the command: 

    COMPILE_SWITCH=sysrs FNAME="abro" \
                     SYSRUST_DIR=<path-of-sysrust-compiler-dir> \
                     TIMETIDE_DIR=<path-of-timetide-compiler-dir> make
    • This will produce three files abro.sysrs and abro.cpp and abro.h.
  2. The file abro.sysrs is the intermediate representation of the abro.tt program in a synchronous format.
  3. Files abro.cpp and abro.h are the generated C++ and header, respectively.
  4. The code snippet below shows the part of the generated main function. The generated main function is in namespace abro – always the name of the .tt file.

  5. You need to now write a main.cpp file that provides the input and output functions. 

     1: namespace abro {
 2:   //....
 3:   int main(void) {
 4:     read_inputs();              // provide this function
 5:     //...
 6:     write_outputs();            // provide this function
 7:     pre_eq_curr();
 8:     reset_curr();
 9:     return 0;
10:   }
11: }

  6. The example main.cpp file is below. The file is available in Timetide_tests directory to modify. 

     1: #include "abro.h"
 2: #include <cassert>
 3: #include <iostream>
 4: int mtick() {
 5:   char b = '\0';
 6:   int k = scanf("%c", &b);
 7:   return b;
 8: }
 9: // In this case there are no inputs and outputs to abro.tt
10: void abro::read_inputs() {}
11: void abro::write_outputs() {}
12: 
13: // The main function
14: int main(void) {
15:   while (1) {
16:     // Run the abro program
17:     abro::main();
18:     // Then we are done with the tick.
19:     if (mtick() == 'd')
20:       break;
21:   }
22:   return 0;
23: }

  7. Once we have the main.cpp file we can compile the abro example to a binary using the command: 

    COMPILE_SWITCH=sysrs FNAME="abro" \
                     SYSRUST_DIR=<path-of-sysrust-compiler-dir> \
                     TIMETIDE_DIR=<path-of-timetide-compiler-dir> EXTRA_FLAGS="-DDEBUG" exe
    • Notice the -DDEBUG macro being provided.

  8. We can finally run the program using the command: 

    COMPILE_SWITCH=sysrs FNAME="abro" \
                    SYSRUST_DIR=<path-of-sysrust-compiler-dir> \
                    TIMETIDE_DIR=<path-of-timetide-compiler-dir> EXTRA_FLAGS="-DDEBUG" run

Compiling the abro example (distributed memory) program

The primary idea is to divide the abro program into two parts: (a) abro_sys.tt, which captures the system waiting for the data to arrive from input channels, and (b) the environment abro_env.tt that produces the inputs for the system.

  1. In this case we will write two different modules/files abro_sys.tt and abro_env.tt. These individual modules are presented below: 

     1: /* abro_sys.tt */
 2: main(input int A, input int B, input int R, output int O) {
 3:   for (int i:1:99) {
 4:          abort(R) {
 5:            /* Wait for A and B in parallel */
 6:            {run waitA(A);}<>{run waitB(B);};
 7:            send O(100);         /* send O once both are received */
 8:            /* abort when R is present and restart */
 9:            task(period=1,duration=0,offset=0){;} /* halt */
10:          }
11:   } /* loop back */
12: };
13: waitA(input int A) {abort(A) {task(period=1,duration=0,offset=0){;}}};
14: waitB(input int B) {abort(B) {task(period=1,duration=0,offset=0){;}}}

    1: /* abro_env.tt */
2: main(output int A, output int B, output int R) {
3:   {send A(1);}<>{send B(1);};
4:   send R(1);
5: }
  2. Important point to notice:
    • Each of these modules are written in their individual files abro_sys.tt and abro_env.tt.
    • Each module has an individual main function/reaction.
    • The main function takes inputs and outputs as required.

  3. In order to compile these individual modules we execute the following command: 

    COMPILE_SWITCH=sysrs FNAME="abro_sys abro_env" \
                          SYSRUST_DIR=<path-of-sysrust-compiler-dir> \
                          TIMETIDE_DIR=<path-of-timetide-compiler-dir> make \
                          EXTRA_FLAGS="-DDEBUG"
    • Notice that we specify the individual module names in FNAME environment variable for compiling both modules together.
  4. Upon success we get files: abro_sys.sysrs, abro_env.sysrs, abro_sys.cpp, abro_sys.h, abro_env.cpp, and finally abro_env.h.
  5. We now need to provide the input and output read write functions: abro_sys::read_inputs, abro_sys::write_outputs, abro_env::read_inputs, and abro_env::write_outputs in the main.cpp file for these individual modules to communicate with each other.
  6. These functions are presented below:

    • The abro_env::write_outputs function: 

       1: // This function serializes and then send the data from output port to
 2: // the input port.
 3: template <typename In, typename Out, typename Q>
 4: void In2Out(In *in, Out *out, Q *q) {Qpush(q, (void *)in, out);}
 5: // We always write the current value from one to the previous value of
 6: // the next.
 7: void abro_env::write_outputs() {
 8:          // Then we write the outputs for the tick done.
 9:          In2Out(abro_env::A_curr_ptr, abro_sys::A_prev_ptr, abro_sys::qA_ptr);
10:          In2Out(abro_env::B_curr_ptr, abro_sys::B_prev_ptr, abro_sys::qB_ptr);
11:          In2Out(abro_env::R_curr_ptr, abro_sys::R_prev_ptr, abro_sys::qR_ptr);
12: }
    • Function abro_env::write_outputs (Line 7) passes the outputs from module abro_env to module abro_sys.
    • This function binds the output from abro_env module to abro_sys module.
    • The output channel A in module abro_env (specifically abro_env::A_curr_ptr) is bound to input channel A in module abro_sys (specifically abro_env::A_prev_ptr). Same for channels B and R.
    • Function In2Out achieves this binding via queue on the receiving side – in this case module abro_sys.
    • The third argument in function In2Out (Line 4) is this receiving queue for each channel.
    • The size of the queue is determined by the delay \(\delta\) connecting the channels, and is determined by the system designer.
    • The Qpush function (Line 4) is provided in the library (which we will see later) for pushing into the queue of the receiver.

    • The abro_env::read_inputs function: 

      // No inputs needed, because main module in abro_env.tt has no inputs
void abro_env::read_inputs() {}
  7. The channels being used to connect the two modules along with the queue functions are provided in files:
    • abro_env.h, abro_sys.h, includes/lib.h, and includes/mQueue.h in the Timetide_tests directory.
    • The abro_env.h and abro_sys.h headers are generated by the compiler and are specific to the modules being compiled.
    • The headers in the includes folder form the part of the library used to compile distributed memory systems.

  8. The important parts of the generated abro_env.h header is shown below: 

     1: namespace abro_env {
 2:   // ....
 3:   struct signal_A;
 4:   // The below are the opaque pointers used for communication
 5:   extern signal_A *A_curr_ptr, *A_prev_ptr;
 6:   // A_curr_ptr is a pointer to status/value of channel A for current tick.
 7:   // A_prev_ptr is a pointer to status/value of channel A for previous tick.
 8:   struct signal_B;
 9:   extern signal_B *B_curr_ptr, *B_prev_ptr;
10: 
11:   struct signal_R;
12:   extern signal_R *R_curr_ptr, *R_prev_ptr;
13: 
14:   // Templates for opaque channel pointers
15:   template <typename T> size_t get_sizeof_value(T *);
16:   template <typename T> size_t get_sizeof_status(T *);
17:   template <typename T> bool get_status(T *);
18:   template <typename T> void get_value(void *dest, T *signal);
19:   template <typename T> void set_status(T *signal, unsigned char v);
20:   template <typename T> void set_value(T *signal, void *src);
21:   template <typename T, typename S> void Qpush(T *, void *, S *);
22:   template <typename T, typename S> size_t Qpull(T *, S *);
23:   void write_outputs();
24:   void read_inputs();
25: }

  9. The implementation of these header functions is provided in includes/lib.h and is provided below: 

     1: template <typename T>
 2: concept has_status = requires(T* t) {
 3:   {t->status};
 4: };
 5: 
 6: template <typename T>
 7: concept has_value = requires(T *t) {
 8:   { t->value };
 9: };
10: 
11: template <typename T>
12: requires has_value<T>
13: constexpr size_t get_sizeof_value(T *obj) {
14:   return sizeof(obj->value);
15: }
16: 
17: template <typename T>
18: requires has_status<T>
19: constexpr size_t get_sizeof_status(T *obj) {
20:   return sizeof(obj->status);
21: }
22: 
23: template <typename T>
24: requires has_status<T>
25: bool get_status(T *obj) { return obj->status; }
26: 
27: template <typename T>
28: requires has_value<T>
29: void get_value(void *dest, T *obj) {
30:   memcpy(dest, &(obj->value), get_sizeof_value(obj));
31: }
32: 
33: template <typename T>
34: requires has_status<T>
35: void set_status(T *obj, unsigned char v) {
36:   memcpy(&(obj->status), &v, get_sizeof_status(obj));
37: }
38: 
39: template <typename T>
40: requires has_value<T>
41: void set_value(T *obj, void *src) {
42:   memcpy(&(obj->value), src, get_sizeof_value(obj));
43: }
44: 
45: template <typename Q, typename S>
46: requires has_status<S> && has_value<S>
47: void Qpush(Q *q, void *src, S *s) {
48:   constexpr std::size_t size = sizeof(s->status) + sizeof(s->value);
49:   std::size_t size_status = get_sizeof_status(static_cast<S *>(src));
50:   // This array is stack allocated
51:   std::array<unsigned char, size> arr{0};
52:   memcpy(arr.data(), &(static_cast<S *>(src))->status, size_status);
53:   memcpy(arr.data() + size_status, &(static_cast<S *>(src))->value, size - size_status);
54:   q->v->push(std::move(arr));
55: }
56: 
57: template <typename Q, typename S>
58: requires has_status<S> && has_value<S>
59: std::size_t Qpull(Q *q, S *s) {
60:   constexpr std::size_t size = sizeof(s->status) + sizeof(s->value);
61:   using at = std::array<unsigned char, size>;
62:   at *res = q->v->front(); // the pointer to the top of the queue.
63:   s->status = (*res)[0]; // the first value in the returned array
64:                          // pointer
65:   set_value(s, &(*res)[1]);
66:   q->v->pop();
67:   return size;
68: }

  10. The templated library function implementations are specialised for each channel in generated C++ code from the Time tide modules. Example shown below from abro_env.cpp: 

     1: namespace abro_env {
 2:   //...
 3:   #include "lib.h"
 4: 
 5:   // Sig decls
 6:   typedef struct signal_A {
 7:     int value = 0;
 8:     bool status = false;
 9:     // We should never need the combinator operator
10:     std::plus<int> op{};
11:     // tag is for fresh value updates
12:     bool tag = false;
13:   } signal_A;
14:   signal_A A_curr, A_prev;
15:   signal_A *A_curr_ptr = &A_curr;
16:   signal_A *A_prev_ptr = &A_prev;
17:   // Below are template specilisations for working with channel A
18:   template size_t get_sizeof_value(signal_A *);
19:   template size_t get_sizeof_status(signal_A *);
20:   template bool get_status(signal_A *);
21:   template void get_value(void *, signal_A *);
22:   template void set_value(signal_A *, void *);
23:   template void set_status(signal_A *, unsigned char v);
24:   // ......
25: }
    • Notice the template specialisation from (Line 18 – Line 23)

  11. Similarly, the receiving Queue' opaque queue pointers (e.g., abro_sys::qA_ptr) are declared and the Qpush and Qpull template functions specialised in the generated C++ files. See below for a snippet of the generated abro_sys.cpp: 

     1: #include "mQueue.h"
 2: namespace abro_sys {
 3:   // ....
 4: #include "lib.h"
 5:   // ......
 6:   // mQueue is defined in includes/mQueue.h
 7:   using qA_t = mQueue<std::array<unsigned char, sizeof(A_prev_ptr->status) +
 8:                                  sizeof(A_prev_ptr->value)>>;
 9:   // mQueue.h provides a lockfree queue implementation, so it can be used with many threads
10:   qA_t qA(A_DELAY, {});
11:   struct QA {
12:     qA_t *v;
13:     QA(qA_t *p) : v(p) {}
14:   };
15:   QA qA_var(&qA);
16:   QA *qA_ptr = &qA_var;

18:   template size_t Qpull(QA *, signal_A *);       // template specilisation
19:   // ....
20: }
  12. Never include standard library headers in includes/lib.h, because it is included in the module' namespace.

  13. The delay \(\delta\) determining the size of the queues is defined in file delays.cpp. For the current example the code is shown below: 

    1: #include "abro_sys.h"
2: int abro_sys::A_DELAY = 4;      // size of queue connecting channel A
3: int abro_sys::B_DELAY = 2;      // size of queue connecting channel B
4: int abro_sys::R_DELAY = 5;      // size of queue connecting channel R

  14. Finally, we can now write the main function that runs the two modules. This function is written in main.cpp and is shown below: 

     1: #include "abro_env.h"
 2: #include "abro_sys.h"
 3: #include <atomic>
 4: #include <cassert>
 5: #include <iostream>
 6: #include <thread>
 7: void abro_run(std::atomic<int> &&barrier = 0) {
 8:   std::cout << "running two threads\n";
 9:   auto env = std::thread([&barrier]() {
10:     while (barrier != 1) {
11:       std::this_thread::yield();
12:     }
13:     barrier--;
14:     // First we read from the buffer for all inputs
15:     std::cout << "Running the abro env \n";
16:     abro_env::main();
17:     std::cout << "Done running the abro env\n";
18:   });
19: 
20:   auto sys = std::thread([&barrier]() {
21:     while (barrier != 2) {
22:       std::this_thread::yield();
23:     }
24:     barrier--;
25:     std::cout << "Running the abro sys\n";
26:     abro_sys::main();
27:     std::cout << "Done running the abro sys\n";
28:   });
29:   // XXX: you can change this to change the order of execution.
30:   // barrier = 2, makes sys run first.
31:   // barrier = 1, makes the env run first.
32:   // think of barrier as a token that is passed to threads to make then
33:   // execute an LSN.
34:   barrier = 2;
35:   while (barrier != 0)
36:     ; // spin until both the modules/threads are done -- the barrier
37:   env.join();
38:   sys.join();
39: }
40: int main(void) {
41:   while (1) {
42:     abro_run();
43:     // Then we are done with the tick.
44:     if (mtick() == 'd')
45:       break;
46:   }
47:   return 0;
48: }
    • Notice that we are using threads to run the two modules.
    • The first thread runs abro_env::main (Line 9)
    • The second thread runs abro_sys::main (Line 20)
    • The two threads execute one tick each before reaching a barrier (Line 35)
    • It is essential that all modules in the Time tide program execute using barriers, because they all need to be synchronised.
    • Every module reaching the barrier indicates the end of a global tick.

  15. Once all the required files and functions are in place we can generate the binary to execute using the command: 

    COMPILE_SWITCH=sysrs FNAME="abro_sys abro_env" \
                          SYSRUST_DIR=<path-of-sysrust-compiler-dir> \
                          TIMETIDE_DIR=<path-of-timetide-compiler-dir> make \
                          EXTRA_FLAGS="-DDEBUG delays.cpp" exe

  16. We can then run the program like so: 

    COMPILE_SWITCH=sysrs FNAME="abro_sys abro_env" \
                          SYSRUST_DIR=<path-of-sysrust-compiler-dir> \
                          TIMETIDE_DIR=<path-of-timetide-compiler-dir> make \
                          EXTRA_FLAGS="-DDEBUG delays.cpp" run
  17. One needs to write a new library header/C++ file to send data across network leveraging includes/lib.h. This can be via sockets, UART, etc.
  18. When the modules are implemented on physically different machines, make sure that modules still execute using barrier synchronisation.

The Time tide language

Types and variables

Primitive types

We only support three signed primitive types listed below 

1: int     a;                      /* This is a comment (int is 4 bytes) */
2: char    a;                      /* char is  1 byte */
3: float   a;                      /* Float is 4 bytes */

User defined structures

User defined types follows the same convention as C. However, no pointers are allowed. Examples follow: 

1: struct Y { int a; float u;};    /* Structure (type Y) with field varibles a and u */
2: struct X {
3:   int yy;                       /* varible yy of type int */
4:   struct Y g;                   /* variable g of type struct Y */
5: }
6: /* Since there are no pointers, one cannot define self refrential structures */
7: struct X {
8:   struct X *m;                  /* not allowed */
9: }

Array types

Arrays are similar (not the same) to those in C. Multidimensional arrays are supported. Example follows: 

1: const int[4] X = {0, 0, 0, 0};  /* constant array, notice it is int[4]  not int X[4]*/
2: const int[2][3] Y = {{0, 0, 0}, {0, 0, 0}} /* 2 x 3 constant array Y -- multidimensional */

Channels

  1. Channels are specific to Time tide and do not exist in C.
  2. Channels carry data.
  3. Channels also have an associated delay \(\delta\).
  4. Channels can be of any primitive, user defined, or array type.
  5. Channel status and values are set using send (Line 8)
  6. Channel status is read using the name of the channel (Line 9)
  7. Channel value is read using ?<channel-name> (Line 11)
  8. Examples follow:
 1: int channel K delay 3;          /* Channel K of type int with delay 3 */
 2: struct Y channel C delay 2;     /* Channel C of type struct Y with delay 2 */
 3: int[100] channel M delay 1;     /* Channel M of type int[100] with delay 1*/
 4: 
 5: /* TODO: Such bunching of channels  are not yet supported */
 6: struct Y[100] channel M[100] delay 7; /* 100 channel M[0]...M[99] each of
 7:                                          type struct Y[100] with delay 7*/
 8: send K(10);                     /* Will be available at receiver after 3 logical ticks */
 9: if (K) {
10:   mut int uu = 0 {              /* Definig a mutable integer uu, scope starts at {*/
11:    uu = ?K              /* Updating the value of uu by reading the value of channel K */
12:   }                     /* scope of variable uu ends */
13:   /* body... */
14:  }

Operators and expressions in Time tide

Operators in Time tide

 1: a + b;                          /* addition */
 2: a - b;                          /* subtraction */
 3: a % b;                          /* Modulus  */
 4: a * b;                          /* Multiply */
 5: a / b;                          /* Divide */
 6: a >> b;                         /* Right shift logical */
 7: a << b;                         /* Logical left shift */
 8: a > b;                          /* Logical greater than */
 9: a < b;                          /* Logical less than */
10: a >= b;                         /* Logical greater than equal */
11: a <= b;                         /* Logical less than equal */
12: a == b;                         /* Logical equality comparison */
13: a != b;                         /* Logical not equal to comparison */
14: a ^^ b;                         /* a to the power of b */
  • The above operators are defined on primitive types only.

Expressions in Time tide

  • Time tide supports the usual expressions as in C.
 1: a + b - c                       /* left to right precedence */
 2: (a + b) * c                     /* Bracket has higher precedence */
 3: a + b * c                       /* Multiply has higher precedence */
 4: a * b / f                       /* Left to right precedence */
 5: (a == b) && (b != c)            /* Logical and expression */
 6: (a == 1) || (b == 1)            /* logical or */
 7: !(a == 0)                       /* logical not */
 8: (int)1.0                        /* casting C style */
 9: 
10: channel int K delay 1;                  /* int type channel */
11: channel int I1 delay 1;                 /* int type channel */
12: extern f(1.0+(float)?K * (float)?I1); /* calling an external C function */
13: 
14: struct X{int a;};
15: struct Y{
16:   float h;
17:   struct X xx;
18: }
19: /* Let yy be a variable of type struct Y */
20:   y.h = (float)7;                       /* assigning to field h */
21:   y.xx = {(int)8.5};                    /* assigned to field xx of type struct X */
22: y.xx.a = (int)8.5;                      /* Not allowed */
23: struct X {
24:   float x;
25:   int[2] uuar;  /* Currently arrays inside structs are not supported */
26: };
  • extern function calls are expressions (not statements)
  • Might need to cast the output or arguments to correct type in extern function calls (Line 12)
  • Only one level of struct fields can be accessed right now (Line 20 – Line 22) – will change in the future
  • Arrays inside struct are currently not supported – will change in the future (Line 25).

  • A more detailed example of calling extern C functions is provided below: 

    /* Intelligent Driver Model car system with external C function call */
extern {                        /* extern function declaration */
  /* The below is the type signature of the function defined in C */
  struct IDMState idm_run(float, float, float, float, float);
};

/* Intelligent Driver Model car state */
struct IDMState {
  float position;
  float velocity;
  float acceleration;
  float gap;
  float lead_position;
  float lead_velocity;
};

/* Intelligent Driver Model car system main function/module */
main(input float lead_position, input float lead_velocity, 
     output float car_position, output float car_velocity,
     output float car_acceleration)
/* The arguments to main are the I/O to this module */
{
  const float dt = 0.1; /* Time step -- constant variable */
  /* Initial car state -- mutable variable */
  mut struct IDMState car = {0.0, 20.0, 0.0, 0.0, 0.0, 0.0} { /* mutable variable */
    /* Wait for fresh lead vehicle data from the input channels */
    if (lead_position && lead_velocity) {
      /* Get current lead vehicle state */
      mut float current_lead_pos = 0.0 { /* mutable variable */
        current_lead_pos = ?lead_position; /* update variable with value of channel */
        mut float current_lead_vel = 0.0 { /* mutable variable */
          current_lead_vel = ?lead_velocity; /* update variable with value of channel */
          /* Call external C IDM function and get struct result */
          /* IDM returns IDMState struct with all updated values */
          /* Run the idm_run function and return stating that it
             succeeded. */
          car =                 /* extern call -- updating car variable*/
          extern idm_run(dt, car.position,
                         car.velocity, current_lead_pos, current_lead_vel);
        }
      }
    }
  }
}
  • Type of extern function declaration and call have to match, else an error will be raised by the compiler.
  • The types of arguments and return type have to match between the extern type signature declaration and its use.
  • extern functions are unique, i.e., there can only be one function f with a given signature. This is requirement of the C programming language.

Statements

Variable declaration and updates

  1. There are two types of variables const and mutable.
  2. Constant variables cannot be modified.
  3. Mutable variables are like normal C style variables.
  4. Variables should always be initialised with literal constants (not expressions).
  5. Shadowing variable names is not allowed.
  6. Examples follow:
 1: const int U11 = 1;              /* constant variabel U11 cannot change */
 2: const int T = 10;               /* constant variable T cannot change */
 3: mut float hh = 1.0 {
 4:   /* ..... body/scope of mutable variable */
 5: } /* Mutable variable hh with its scope defined by {} */
 6: mut float f;                    /* error, variable needs to be initialised */
 7: float f;                        /* error, do not know whether this is const or mutable */
 8: 
 9: 
10: /* Define a structure */
11: struct X {
12:   float x;                      /* field x */
13: };
14: 
15: /* Define another structure */
16: struct Y {
17:   int a;                        /* field a */
18:   struct X m;                   /* field m */
19: };
20: 
21: const int[2] U1 = {1, 2};       /* Constant array U1 of type int[2] with elements 1 and 2*/
22: struct Y channel C delay 2;     /* Channel C of type struct Y with delay 2 */
23: /* A mutable structure */
24: mut struct Y y = {100, {9.6}} {
25:   y.a = U1[0];                  /* Updating the field of y */
26:   /* Send on channel C */
27:   send C(y);                    /* Will be available at receiver after two global ticks */
28: }
29: const int HH = 100;
30: mut int HH = 90 {
31:   /* body */
32: } /* Such shadowing of variable name HH is *not* allowed */

Conditionals

  1. Only if-else blocks are supported.
  2. The semantics are similar (not the same) as C.

  3. Example follows: 

     1: const int K1 = 0;
 2: /* This is getting the value of the channel */
 3: mut int U = 0 {
 4:   U = ?L;                       /* get the value of channel */
 5:   if (?L == 1) {;}              /* check the value of the channel */
 6:   else {U = 100;}
 7: }
 8: if (L) {                        /* check status of channel */
 9:   task(period=1, duration=1, offset=0) {;}
10: }
  4. Major difference to C

    • Checking value of variable/channel requires one to write: 

      1: if(<var-name> == <some-expression>){/*body*/}
2: if(<?channel-name> == <some-expression>){/*body*/}

    • Checking the status of the channel requires one to write: 

      1: if(<channel-name>) {/*body*/}
    • Expressions in conditionals for variables. Using name only implies checking channel status.

Loops

  • Sequential loops
    1. Sequential loops in Time tide are always bounded.
    2. Sequential loops are of two types:
      • Yielding, where every loop iteration produces results in completion of module' tick
      • Instantaneous, where the whole loop gets unrolled.

    3. Example of instantaneous loop follows: 

       1: mut int[4] g = {0, 0, 0, 0} {   /* declare an array */
 2:   /* The below is an instantaneous loop */
 3:   for(int i : 0:4) {            /* Loop through the array */
 4:     g[i] = g[i] + i;            /* update the value of the array */
 5:   }
 6: }
 7: /* The above loop is equivalent to */
 8: mut int[4] g = {0, 0, 0, 0} {   /* declare an array */
 9:   g[0] = g[0] + 0;              /* update the value of the array */
10:   g[1] = g[1] + 1;              /* update the value of the array */
11:   g[2] = g[2] + 2;              /* update the value of the array */
12:   g[3] = g[3] + 3;              /* update the value of the array */
13: }

    4. Example of yielding (non-instantaneous) loop follows: 

       1: mut int[4] X = {0, 0, 0, 0} {   /* declare array X */
 2:   mut int v = 0 {               /* declare a mutable variable */
 3:     for(int i:0:4) {            /* Iterate through the loop */
 4:       /* Define a task, which performs its body every 1 logical tick */
 5:       task(period=1, duration=0, offset=0) {
 6:         v = v + i;              /* increment v every tick */
 7:         X[i] = v + i;           /* update X[i] every tick */
 8:       }                         
 9:     }/* break once iterations are done */
10:   }
11: }
      • The above code will update the value of X' elements once/logical tick of the module.
      • This is different to instantaneous loop where all of X' elements get updated in the same logical tick of the module.
  • Parallel loops
    1. Parallel loops execute statement in synchronous parallel – update values together.
    2. Parallel loops in Time tide are always bounded.
    3. Parallel loops are of two types:
      • Yielding, where every loop iteration produces results in completion of module' tick
      • Instantaneous, where the whole loop gets unrolled.
    4. Parallel loops should work on disjoint memory locations.

    5. Examples follow: 

       1: mut int[4] g = {0, 0, 0, 0} {   /* start of a parallel loop */
 2:     /* Instantaneous parallel loop */
 3:     par(int i : 0:4) {
 4:         g[i] = g[i] + i;                /* working on disjoint memory */
 5:     } /* end of a parallel loop */
 6: }
 7: mut int[4] g = {0, 0, 0, 0} { 
 8:     /* Non instantaneous parallel loop */
 9:     par(int i : 0:4) {
10:         task(period=1, duration=0, offset=0) { /* update g in parallel and complete tick */
11:             g[i] = g[i] + i;                   /* working on disjoint memory */
12:         }
13:     }
14: }
15: 
16: /* Incorrect parallel loop */
17: mut int v1 = 0 { 
18:     par(int j : 0:10) {
19:         v1 = v1 + j;            /* updating the same location in parallel */
20:     }
21: }
22: /* Incorrect parallel loop */
23: int channel FH delay 1;
24: mut int v1 = 0 {
25:     par(int j : 0:10) {
26:         send FH(10);            /* sending in parallel is not allowed */
27:     }
28: }
29: 
30: /* Potentially incorrect parallel loop */
31: mut int[4] g = {0, 0, 0, 0} {
32:   par(int i : 0:4) {
33:    g[i] = g[i-1] + i; /* Loop carried dependency */
34:   }
35: }
      • Line 4, in the code above, updates different memory locations of array g in parallel.
      • Loop on Line 1 runs instantaneously without generating a logical tick.
      • Loop on Line 9 updates the disjoint memory locations and then completes a logical tick.
      • Loop on Line 18 is incorrect since it is updating the same memory location v1 in parallel.
      • Loop on Line 25 is incorrect since it is sending on the same channel FH in parallel.
      • Loop on Line 32 is potentially incorrect, because the loop carried dependency on Line 33. It is unclear what the programmer intends.
      • Incorrect loops, writing to the same memory location, raise a compiler warning.
      • Incorrect loops, sending on channels in parallel, raise a compiler error.
      • Compiler warnings are overestimates, not exact.

The Task construct

  1. Tasks are the primary construct to introduce state (tick) in a Time tide program.
  2. Tasks have a period, which determines how often the body of the task ticks.
  3. Tasks have a duration, which determines how many ticks does the body of the task takes.
  4. Tasks have an offset, which determines a starting offset (in number of logical ticks) the body of the task.
  5. Tasks cannot be nested.
  6. Task bodies have to be instantaneous.
  7. Task period has to be greater than or equal to the duration + offset specified.
  8. The channel delays interact in complex ways with the task period, please follow compiler output when sending on channels inside tasks.

  9. Examples follow: 

     1: motorA (output int MOTOR_A_SPEED,
 2:         input int LIGHT_LOW_1,
 3:         input int LIGHT_HIGH_1) {
 4:   for(int i: 0:99) {
 5:     send MOTOR_A_SPEED(100);
 6:     abort(LIGHT_LOW_1) {                    /* preempt when LIGHT_LOW_1 channel status is true */
 7:       task(period=1,duration=0,offset=0){;} /* self loop on state */
 8:     }
 9:     send MOTOR_A_SPEED(20);
10:     abort(LIGHT_HIGH_1) {                    /* preempt when LIGHT_HIGH_1 channel status is true */
11:       task(period=1,duration=0,offset=0) {;} /* self loop on state */
12:     }
13:   }
14: };
15: int channel C delay 3;
16: /* Incorrect channel send, delay-1 != task' period */
17: task(period=3, duration=2, offset=1) {
18:   send C(1);
19: } /* compiler will raise an error in the above case */
20: 
21: /* Incorrect task, because period is not >= duration + offset */
22: task(period=2, duration=2, offset=1) {
23:   send C(1);
24: }

The Abort construct

  1. abort(logical-expression){/* body */} is the way to preempt a halted system.

  2. Example follows: 

    1: mut int yt = 100 {
2:   /* We check for the value of channel C and status both*/
3:   abort(?C == yt || C) {             /* preempt when the logical expression is true */
4:     task(period=2, duration=1, offset=1) {;} /* halted forever */
5:   }
6:   yt = yt + ?C;                 /* make progress here only after preeemption */
7: }

The Synchronous parallel construct

  1. {/*thread body 1*/}<>{/*thread body 2*/}<>{/*thread body 3*/}... is the way to execute multiple bodies in parallel.
  2. All requirements of par loop also apply to synchronous parallel execution.

  3. Examples follow: 

     1: h(input const int T, output int I, input int L)
 2: {
 3:   /* const int U11 = 100; */
 4:   {run m(T, I);} <> {run m1(T, L);} /* m and m1 are some modules/functions */
 5: };
 6: 
 7: /* Another example -- snippet from the abro example */
 8: main() {
 9:   int channel A delay 1;
10:   int channel B delay 1;
11:   int channel R delay 2;
12:   int channel O delay 1;
13:   {
14:     for (int i:1:99) {
15:       abort(R) {
16:         /* Wait for A and B in parallel */
17:         {run waitA(A);}<>{run waitB(B);};
18:         send O(100);            /* send O once both are received */
19:         /* abort when R is present and restart */
20:         task(period=1,duration=0,offset=0){;} /* halt */
21:       }
22:     } /* loop back */
23:   }<> /* synchronous parallel */
24:     {/*The environment*/
25:       {send A(1);}<>{send B(1);}; /* another sychronous parallel */
26:     }
27: }

Function/module definition

  1. Function/module definition is similar to C.
  2. Functions can take as arguments constant and channels.
  3. The programmer needs to specify whether this is an input or output channel or constant.
  4. main is the entry point of the program.
  5. Structures and functions are declared at the same level inside a .tt file, i.e., one cannot define structures inside functions.
  6. Functions and structures are departed with a semicolon.

  7. A robot motor controller example follows: 

     1: main () {
 2:   /* declare the required channels and constants */
 3:   int channel LIGHT_LOW_1 delay 1;
 4:   int channel LIGHT_HIGH_1 delay 1;
 5:   int channel LIGHT_LOW_2 delay 1;
 6:   int channel LIGHT_HIGH_2 delay 1;
 7: 
 8:   /* Output/Input channels to the observer? */

17:   int channel SET_LIGHT_2_THRESHHOLD delay 1;

20:   send SET_LIGHT_2_THRESHHOLD(80);
21:   send MOTOR_A_DIR(0);
22:   send MOTOR_C_DIR(0);
23: 
24:   /* Run the modules */
25:   {run motorA(MOTOR_A_SPEED, LIGHT_LOW_1, LIGHT_HIGH_1);}
26:     <> {run motorC( MOTOR_C_SPEED, LIGHT_LOW_2, LIGHT_HIGH_2);}
